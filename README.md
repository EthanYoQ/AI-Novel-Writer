# AI 小说作家

**English README**：[README_en.md](README_en.md) · **下载**：[Windows v0.2.0 zip](https://github.com/EthanYoQ/AI-Novel-Writer/releases/tag/v0.2.0) · **许可证**：[GPL-3.0](LICENSE)

> 本地优先的 AI 长篇小说创作桌面工作台。你的故事，你的数据，你的模型。

写小说最怕的不是写得慢——是**写到第 30 章的时候，第 5 章埋的伏笔被忘了**，主角的性格悄悄漂移，配角突然消失，AI 越写越像在放飞自我。**AI 小说作家**把"前提 → 角色 → 世界观 → 章节蓝图 → 草稿 → 审稿 → 修稿 → 定稿"做成一条带记忆的生产线，让 AI 每次只写本章、每次都看蓝图、每章都过审稿。本地运行，**支持本地模型和任何 OpenAI 兼容端点**。

---

## 一句话三连

- 🔓 **本地模型 + 不受限的创作** —— 接入 Ollama / LM Studio / vLLM / KoboldCpp 等本地服务，配合 abliterated / uncensored 权重（如 Qwen3 14B Q4），绕过云端 API 的内容审查。
- 🎨 **拆解一本小说，仿写它的味道** —— 丢一本你喜欢的 TXT / Markdown 进去，自动反推章节大纲、提取角色卡、生成文风约束，整套配方交给 AI 接着写。
- 🧩 **角色/大纲/蓝图互相咬合** —— 改了一个角色，所有相关章节蓝图自动标记；写了一章新稿，所有引用这条蓝图的设定自动回流。**角色漂移、情节走样、前后矛盾，靠结构而不是靠 prompt 解决**。

## 凭什么不让 AI 写崩你的长篇小说

### 1. 本地模型，零审查

云端 API 的内容安全策略不会因为你写的是虚构小说就放行——**你写的主角在小说里受伤，模型拒答**。本地模型没有这个问题：

| 接入方式 | 适合场景 |
| --- | --- |
| **Ollama**（推荐） | 一行命令跑 `qwen3:14b-abliterated` 之类已解锁权重的模型 |
| LM Studio / vLLM / KoboldCpp | 本地推理服务，OpenAI 兼容协议直连 |
| OpenAI / DeepSeek / Gemini | 不想本机跑模型时的云端备选 |
| 自定义 OpenAI-compatible 端点 | 公司内网代理、其他开源推理服务 |

内置提示词针对 **Qwen3 14B Q4 级别**模型做过收敛：短指令、明确字段、结构化输出、分阶段生成、降低重复、只写本章、保留角色和蓝图约束。

> 配置文件 `~/.vela/config.json` 里的 `defaultModelId` 指向你的本地模型名即可。小说项目数据全部落在本机 SQLite 和项目目录里，**不联网也能写**。

### 2. 拆解仿写：把别人的笔变成你的

随便选一本你喜欢的小说（TXT / Markdown），丢进"拆解仿写"：

1. 自动分章
2. 反推全局设定（题材、视角、节奏）
3. 提取每个角色的卡（背景、动机、说话习惯）
4. 生成每一章的蓝图（目的、出场、关键事件、悬念钩子）
5. **输出一份文风约束文档**——之后你写自己的小说时，可以把这份约束挂到 AI 的 prompt 里

这意味着你**不是从零教 AI 怎么写**，而是让 AI 从范本里把"语感"学过去。

### 3. 结构化记忆：让 AI 记住第 5 章埋的伏笔

长篇小说最大的工程难题不是字数，是**一致性**。AI 小说作家的工程方案是**把所有创作资产变成可被引用的结构**：

```
项目配置
  ├── 故事前提
  ├── 角色图谱（每个角色的卡 + 关系图）
  ├── 世界观
  ├── 情节大纲
  ├── 章节蓝图（每章：目的 / 视角 / 出场角色 / 关键事件 / 钩子）
  ├── 知识库（TXT/MD 文档，向量 + FTS 全文检索）
  ├── 参考小说拆解产物
  └── 草稿 → 审稿报告 → 修稿 → 定稿
```

- **改一个角色的设定** → 涉及该角色的所有蓝图自动标红提醒
- **写完一章新稿** → 知识库自动入库，后续章节可检索
- **生成草稿前** → 拼装：本章蓝图 + 相关角色卡 + 世界观片段 + 文风约束 + 历史摘要
- **草稿完成后** → 自动生成结构化**审稿报告**（角色状态、时间线、设定冲突、章内逻辑），再用报告驱动**修稿版本**

AI 永远只看得到它需要看的东西，但**不会忘记关键的设定**。

## 核心能力

- 📚 **长篇项目管理**：项目配置、故事前提、世界观、角色卡、章节蓝图、草稿、修稿、审稿报告、定稿，全部落在本机项目目录
- 🧭 **故事架构**：按步骤生成前提、角色图谱、世界观、情节大纲
- 🗂️ **章节蓝图**：每章独立的目的、视角、出场、关键事件、悬念钩子
- ✍️ **正文生成与修稿**：读取蓝图、角色卡、世界观、文风约束、历史摘要，生成草稿、修稿、定稿
- 🧐 **审稿驱动修复**：先生成结构化审稿报告，再用报告驱动修稿
- 📖 **知识库检索**：TXT / Markdown 文档 + 文件夹导入；配了 embedding 走向量检索，没配走 SQLite FTS
- 🧬 **参考小说拆解**：导入小说 → 拆章 → 反推全局 → 提取角色 → 生成蓝图 → 输出文风约束
- 🔌 **模型自由**：OpenAI 兼容协议 + Gemini 协议；本地 / 云端随意切
- 🌐 **中英界面切换**：跟随系统，手动选择后持久化

## 创作工作流

```text
  创作脑洞 / 项目配置
        │
        ▼
  故事前提 → 角色图谱 → 世界观 → 情节大纲
        │
        ▼
   角色卡 / 关系图谱 / 状态追踪
        │
        ▼
      章节蓝图
        │
        ▼
  正文草稿 → 审稿报告 → 修稿版本 → 定稿
        │
        ▼
  知识库回写、角色状态更新、后续章节继续读取
```

## 30 秒跑起来（Ollama 本地模型）

```bash
# 1) 拉一个本地模型（Qwen3 14B 量化版，6GB 显存即可）
ollama pull qwen3:14b

# 2) 在 AI 小说作家的"模型设置"里填：
#    Provider:        custom
#    Protocol:        OpenAI-compatible
#    Base URL:        http://127.0.0.1:11434/v1
#    API key:         ollama
#    Model:           qwen3:14b
#    （或者用社区已解锁的 abliterated 权重，按需自取）

# 3) 新建项目 → 写一句话前提 → 让 AI 依次生成角色 / 世界观 / 蓝图 → 开始写第一章
```

## Windows 安装

发布形式是一个 zip 压缩文件夹：

```text
AI-Novel-Writer-0.2.0-windows-x64.zip
└─ AI-Novel-Writer/
   ├─ AI小说作家.exe
   ├─ resources/
   └─ Electron runtime files...
```

1. 从 [GitHub Releases](https://github.com/EthanYoQ/AI-Novel-Writer/releases/tag/v0.2.0) 下载 `AI-Novel-Writer-0.2.0-windows-x64.zip`
2. **完整解压**到任意目录
3. 双击 `AI小说作家.exe` 启动

> ⚠️ 不要直接在压缩包内双击 EXE——Electron 的相对路径会找不到 `resources/`。解压后运行。

## 数据安全

不要提交到仓库：

- 本机小说项目、导入的参考小说全文
- 生成的草稿 / 修稿 / 定稿正文
- `.env`、API key、本地模型配置
- `release/`、`dist/`、`dist-electron/`、`node_modules/`

项目数据默认保存在本机项目目录和 SQLite 数据库中。如果你配置了外部 API 或本地模型服务，提示词和上下文会发送给对应端点。
